# ADR-006: IAM Strategy

- **Status:** Accepted
- **Date:** 2026-06-20
- **Deciders:** Architecture Team

## Context

Identity and Access Management (IAM) is the foundational module on which all other modules depend. It must provide:
- **Authentication**: user login, password management, MFA, session management, API key authentication for machine-to-machine communication.
- **Authorization**: role-based access control with fine-grained permissions, per-tenant role customization.
- **Identity Management**: user lifecycle (create, update, deactivate, delete), profile management, identity verification flags.
- **Tenant Management**: tenant provisioning, suspension, settings, and lifecycle.
- **SSO/Identity Federation**: integration with external identity providers (LDAP, Active Directory, SAML) for enterprise customers.
- **Token Management**: JWT issuance, validation, refresh token rotation, token revocation.

Two approaches were evaluated:
1. **Embedded IAM** — The IAM module handles all authentication and authorization internally. Passwords are hashed locally, tokens are issued by the IAM module, RBAC is enforced by the IAM module's API. Keycloak is not used.
2. **External IAM with Keycloak** — Keycloak is the identity provider and authorization server. The IAM module manages tenant and role metadata and synchronizes users and roles to Keycloak. Authentication flows go through Keycloak. Token validation uses Keycloak's JWKS endpoint.

Additionally, authorization models were evaluated:
1. **RBAC only** — Roles are assigned to users; permissions are derived from roles.
2. **RBAC + ABAC (Attribute-Based Access Control)** — Access decisions consider not only roles but also user attributes (department, location), resource attributes (tenant, sensitivity), and environmental attributes (time of day, IP range).

## Decision

We will use **Keycloak as the external IAM provider with RBAC + ABAC authorization**.

Key specifics:
- **Authentication**: Keycloak as the OpenID Connect provider. Authorization code flow with PKCE for browser-based authentication. Client credentials flow for API clients and background services. JWT access tokens (RS256 signed, 15-minute TTL) with refresh token rotation (7-day TTL).
- **Identity Federation**: Keycloak's identity brokering for LDAP/Active Directory integration. First-time login creates a linked local Keycloak user. User attributes are synced from the external provider on each login.
- **Role Management**: Roles are defined in the IAM module (canonical source) and synchronized to Keycloak via the Admin REST API. The IAM module manages role-permission mappings; Keycloak manages user-role assignments for token claims.
- **Authorization**: RBAC with ABAC extensions. Base permissions are role-derived. ABAC rules can further restrict access based on attributes (e.g., "support agents can only view customers in their assigned region", "billing admins can only write invoices for their tenant's billing accounts"). ABAC rules are evaluated in the IAM module via a policy engine, not in Keycloak (to avoid complex Keycloak policy configuration).
- **Token Contents**: Access tokens contain `sub` (user ID), `tenant_id`, `roles` (list of role names), `email`, and `name`. Permissions are resolved on demand by the IAM module and cached in Redis — they are not embedded in the JWT to keep token size small.
- **API Key Authentication**: API clients authenticate via `X-API-Key` header. The IAM module hashes the API key (HMAC-SHA256) and validates against stored hashes. API keys are permission-scoped and rate-limited.
- **Password Policies**: Configured per tenant. Enforced at the IAM module level during password changes. Keycloak's built-in password policies (length, complexity, history, expiry) are also configured as a secondary enforcement layer.

## Consequences

### Positive
- **Keycloak provides enterprise-grade authentication out of the box** — password hashing (bcrypt/PBKDF2), brute-force detection, session management, MFA (TOTP, SMS, email), and OIDC compliance are all handled without custom security code. The platform inherits Keycloak's security audit history and community scrutiny.
- **SSO capability from day one** — Keycloak's identity brokering allows the platform to integrate with any SAML 2.0 or OpenID Connect identity provider. Enterprise customers can federate their existing Active Directory or LDAP, reducing password fatigue and support tickets for password resets.
- **RBAC + ABAC provides flexible authorization** — simple use cases (role-based access) are configured quickly. Complex use cases (region-restricted support agents, time-based provisioning access) can be expressed as ABAC rules without code changes. The policy engine evaluates rules at runtime based on request context.
- **Clean separation of concerns** — the IAM module focuses on business identity management (tenants, roles, permissions, user lifecycle) while Keycloak focuses on authentication protocol compliance, token management, and federation. Either can be modified independently.
- **IAM → Keycloak sync is unidirectional** — the IAM module pushes changes to Keycloak. Keycloak never writes back to IAM. This avoids bidirectional sync conflicts and simplifies the data ownership model.
- **Token validation is stateless** — Keycloak's JWKS endpoint provides public keys for token signature verification. The API Gateway validates tokens using the public key without calling Keycloak on every request (cached JWKS, rotated hourly). This enables high-throughput token validation without adding latency.

### Negative
- **Keycloak operational burden** — requires a dedicated PostgreSQL database, JVM heap tuning (typically 1-2 GB minimum), cluster configuration for HA, and regular version upgrades (Keycloak moves fast with breaking changes between major versions). The team must learn Keycloak administration and troubleshooting. Mitigation: containerized Keycloak deployment with health checks and automated backup; use Keycloak's Operator for Kubernetes if available.
- **IAM-Keycloak synchronization is a new failure mode** — a network issue, Keycloak API rate limit, or authentication failure can prevent user creation or role assignment even though the IAM database operation succeeded. Mitigation: async synchronization with retry queue; reconciliation job that detects and fixes drift every 15 minutes; health check endpoint that reports sync status.
- **ABAC policy engine adds complexity** — implementing ABAC requires a policy definition format, a policy evaluation engine, and tooling for policy testing. Developers must understand ABAC concepts and write/test policies. Mitigation: simple expression-based policy language (e.g., `user.region == resource.region`) rather than full rule engine; policy test suite in CI.
- **Two systems to keep in sync for role changes** — roles are created in IAM, synced to Keycloak, and users are assigned roles in Keycloak. If a role is deleted in IAM, the sync must remove it from Keycloak and unassign it from all users. Timing gaps can leave stale role assignments. Mitigation: reconciliation job removes stale role assignments; event-driven role deletion triggers immediate Keycloak cleanup.
- **Keycloak version upgrades can break sync** — Keycloak's Admin REST API changes between major versions. An IAM sync module built for Keycloak 24.x may not work with Keycloak 25.x. Mitigation: version-pinned Keycloak deployment; upgrade testing in staging before production; abstraction layer over Keycloak Admin API to isolate version-specific code.
- **No offline authentication capability** — if Keycloak is unreachable, no user can authenticate (existing sessions with valid JWTs can still be used until token expiry, but new logins and token refreshes fail). Mitigation: Keycloak high-availability cluster; short token TTLs trade off availability for security; consider offline session tokens for critical operational accounts.

### Neutral
- **MFA configuration is per-tenant** — some tenants may require MFA, others may not. Each tenant's settings in the IAM module configure the MFA requirement. Keycloak enforces the requirement at login time. The IAM module must communicate MFA settings to Keycloak during tenant provisioning and update.
- **Passwordless authentication is not yet supported** — the initial version supports password + MFA. Future versions may add WebAuthn/passkeys, magic link, or OTP-based passwordless login. Keycloak supports these flows, so adding them later requires IAM module changes but no infrastructure changes.
