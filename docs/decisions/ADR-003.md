# ADR-003: Security Strategy

- **Status:** Accepted
- **Date:** 2026-06-20
- **Deciders:** Architecture Team

## Context

The OSS/BSS Platform is a high-value target: it handles customer PII (national IDs, addresses, payment instruments), billing data (invoices, payment history), network control (provisioning commands, device credentials), and tenant administration.

The platform must satisfy these security requirements:
- All API endpoints authenticated except public health/registration endpoints
- Role-based access control (RBAC) with fine-grained permissions down to the API endpoint + HTTP method level
- Tenant isolation with no cross-tenant data access (see ADR-002)
- Audit logging for all state-changing operations (see master architecture principle 3.12)
- OWASP Top 10 compliance
- Secure communication (TLS 1.3) for all inter-service and external communication
- Secrets management (no credentials in configuration files or environment variables)
- Machine-to-machine authentication for API clients and automated processes

Three security architecture approaches were evaluated:
1. **Embedded IAM** — Authentication and authorization handled entirely within the platform's IAM module with local user stores and JWT issuance.
2. **External IAM (Keycloak)** — Keycloak as the centralized identity provider and authorization server. IAM module handles tenant and role management; Keycloak handles authentication, token issuance, and token validation.
3. **Cloud IAM (Auth0, Okta)** — Third-party IDaaS provider. Not viable for Yemen market due to internet reliability concerns, data sovereignty, and cost.

## Decision

We will adopt **External IAM with Keycloak** as the core of a defense-in-depth security strategy.

Key specifics:
- **Authentication**: Keycloak as the OpenID Connect provider. Authorization code flow with PKCE for human users. Client credentials flow for machine-to-machine.
- **Authorization**: RBAC with fine-grained permissions stored in the IAM module's database. Permissions follow the format `{module}:{resource}:{action}` (e.g., `billing:invoice:read`). Keycloak roles are synchronized from IAM.
- **API Gateway**: YARP-based reverse gateway that validates every incoming JWT/API key, extracts tenant context, and enforces authentication before routing to module endpoints.
- **OWASP Compliance**: Input validation (FluentValidation on every command), parameterized queries (EF Core + Dapper), CSP headers, anti-forgery tokens for state-changing operations, rate limiting on authentication endpoints.
- **Secrets Management**: HashiCorp Vault or Azure Key Vault for secrets at rest and in transit. No secrets in config files.
- **Defense in Depth**: Multi-tenant RLS, per-request authorization, audit logging, network segmentation, TLS everywhere.

## Consequences

### Positive
- **Keycloak provides battle-tested authentication** — password policies, MFA (TOTP, SMS), brute-force detection, session management, and token lifecycle are handled by a mature product rather than custom code. Avoids the security risks of implementing authentication from scratch.
- **OIDC compliance enables SSO** — the platform can integrate with existing corporate identity providers (LDAP, Active Directory, SAML 2.0 IdPs) through Keycloak's identity brokering. Future customers can federate their existing identity infrastructure.
- **API Gateway provides a single enforcement point** — authentication validation, rate limiting, and request logging are centralized rather than duplicated across 19 modules. Security policies can be updated in one place.
- **Fine-grained permissions enable flexibility** — the RBAC model allows per-tenant role customization. A tenant can define custom roles (e.g., "Junior Billing Analyst" with read-only billing + read-only customer data) without platform-wide changes.
- **Audit trail is comprehensive** — every authenticated request can be correlated to an actor, tenant, and action. Forensic investigation of security incidents is supported by immutable audit logs.
- **Suitable for Yemen market** — on-premise Keycloak deployment works without reliable internet connectivity. No dependency on external identity providers.

### Negative
- **Keycloak operational complexity** — Keycloak requires its own database (PostgreSQL or MySQL), JVM tuning, clustering configuration for high availability, and regular version upgrades. The operations team must be trained on Keycloak administration. Mitigation: containerized Keycloak deployment with automated backup and health monitoring.
- **IAM-Keycloak synchronization introduces failure mode** — the IAM module is the source of truth for users, roles, and permissions. Changes must be synchronized to Keycloak. If synchronization fails, users may exist in IAM but not be able to authenticate, or roles may be out of date. Mitigation: unidirectional sync (IAM → Keycloak) with retry and dead-letter queue; reconciliation job runs every 15 minutes; authentication does not block on sync (uses last-known-good state).
- **JWT token size may be large** — if permission claims are embedded in the JWT, the token can exceed HTTP header size limits for gateways and reverse proxies. Mitigation: permissions are not embedded in the JWT; the token contains only `sub`, `tenant_id`, `roles`. The API Gateway calls the IAM module to resolve permissions on each request (with caching).
- **API Gateway is a single point of failure** — if the Gateway goes down, all external API access is blocked. Mitigation: multiple Gateway instances behind a load balancer; health-check-based routing; Gateway is stateless and can be horizontally scaled.
- **Certificate management burden** — mTLS for network device integration requires certificate issuance, rotation, and revocation for potentially hundreds of network elements. Mitigation: short-lived certificates with automated renewal; certificate management integrated with provisioning workflows.

### Neutral
- **Zero-trust architecture requires cultural adoption** — the security model assumes no trusted network boundary. Every request must be authenticated and authorized, even requests between modules within the same process. This requires developers to think about authorization on every operation, not just external-facing endpoints.
- **Security audits become a regular process** — penetration testing, dependency vulnerability scanning, and compliance audits must be scheduled. OWASP dependency-check runs in CI. Security findings are treated as blocking bugs.
