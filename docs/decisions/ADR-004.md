# ADR-004: Event-Driven Architecture

- **Status:** Accepted
- **Date:** 2026-06-20
- **Deciders:** Architecture Team

## Context

The platform has 19 bounded contexts that must communicate without tight coupling. Many business processes span multiple modules — an order fulfillment event must trigger subscription activation, provisioning jobs, billing start, and notifications. Direct synchronous calls between modules would create coupling, latency chains, and reliability dependencies.

The event-driven architecture must support:
- Reliable cross-module communication without distributed transactions
- At-least-once delivery with idempotent processing
- Event schema evolution over time
- Loose coupling between event publishers and consumers
- Audit trail of all cross-module communication

Three messaging technologies were evaluated:
1. **RabbitMQ** — Mature AMQP broker, supports exchanges, queues, dead-lettering, consumer acknowledgements.
2. **Apache Kafka** — High-throughput event streaming with log-based storage, replay capability, strong partitioning.
3. **Azure Service Bus / AWS SQS/SNS** — Cloud-native message brokers with managed infrastructure. Not viable for on-premise Yemen deployment.

Two reliability patterns were evaluated:
1. **Outbox/Inbox Pattern** — Events are written to a database table within the same transaction as the aggregate change; a background process publishes to the broker. Consumers store received event IDs in an Inbox table for deduplication.
2. **Dual-write with compensating transactions** — Write to database and publish to broker in the same operation with compensating transactions for failures.

## Decision

We will use **RabbitMQ with the Outbox/Inbox pattern, CloudEvents 1.0 standard, and integration events only** (no domain events via message bus).

Key specifics:
- **Message Broker**: RabbitMQ with mirrored queues (quorum queues) for high availability. Topic exchanges for fan-out delivery to multiple consumers.
- **Outbox Pattern**: Domain events that must be published to other modules are persisted to an `outbox` table within the same `SaveChangesAsync` transaction. A background service (`OutboxProcessor`) reads unprocessed records and publishes to RabbitMQ. Records are deleted only after confirmed broker acknowledgement.
- **Inbox Pattern**: Each module's event handler checks the `inbox` table for the incoming `EventId` before processing. If the ID exists, the event is acknowledged and skipped (idempotent deduplication). If new, the event is processed within a database transaction that records the EventId.
- **CloudEvents 1.0**: All integration events follow the CloudEvents specification with `specversion`, `type`, `source`, `id`, `time`, `datacontenttype`, and `data` fields. Event schemas are versioned (e.g., `billing.invoice.generated.v1`).
- **Integration Events Only**: Domain events (internal to a module) are published via MediatR in-process. Only integration events (representing business occurrences that other modules need to react to) are published to RabbitMQ via the Outbox.
- **Event Schema Registry**: Event schemas are defined as C# record types in a shared `Contracts` assembly. Breaking changes are prevented by CI checks (new fields only, no removals).

## Consequences

### Positive
- **Reliable delivery without distributed transactions** — the Outbox pattern ensures atomicity of aggregate state change + event publication without two-phase commit (XA transactions). If the broker is down during a save, events persist in the outbox and will be delivered when the broker recovers.
- **Idempotent processing via Inbox** — exactly-once processing semantics even with at-least-once broker delivery. Duplicate events from broker retries, consumer crashes, or network disruptions are safely handled.
- **Loose coupling** — publishers and consumers know only the event schema. A publisher does not know which modules consume its events. New consumers can be added without modifying the publisher.
- **RabbitMQ is mature and on-premise deployable** — well-documented, stable, containerized deployment, supports clustering, monitoring (Prometheus exporter), and management UI. Works in environments with intermittent connectivity.
- **CloudEvents standard ensures interoperability** — future consumers (webhooks, external systems, data lakes) can consume events without understanding platform-specific formats. CloudEvents SDK is available for .NET.
- **Event-driven debugging is supported** — each event is traceable through the Outbox table, broker queue, and Inbox table. Dead-letter queues capture failed events for manual inspection and replay.
- **No domain event leakage** — the clear distinction between domain events (in-process MediatR) and integration events (RabbitMQ) prevents internal implementation details from leaking across module boundaries.

### Negative
- **Operational complexity of RabbitMQ** — requires cluster configuration, queue management, consumer monitoring, dead-letter handling, and broker upgrades. The operations team must be trained on RabbitMQ administration. Mitigation: containerized deployment with HAProxy for load balancing; monitoring dashboards for queue depth, consumer lag, and dead-letter count.
- **Eventually consistent data across modules** — when an order is submitted, the subscription module does not see it until the integration event is processed. The delay is typically sub-second but introduces complexity for business processes that need immediate visibility. Mitigation: synchronous API queries (through API Gateway) for real-time cross-module data needs; eventual consistency is acceptable for most Telecom processes.
- **Outbox backlog monitoring is essential** — if the OutboxProcessor falls behind, events accumulate in the outbox table. A large backlog means delayed reactions to business events. Mitigation: monitoring alerts on outbox record count > 1000; OutboxProcessor runs with its own connection pool and thread pool to avoid starvation.
- **Event schema evolution is manual** — adding a field to an event requires updating the Contracts assembly, publishing a new schema version, and ensuring all consumers handle both old and new versions. Mitigation: new fields are optional with sensible defaults; consumers use "tolerant reader" pattern (ignore unknown fields); breaking changes require a new event type (e.g., `.v2`).
- **No event replay for schema migration** — unlike Kafka which stores events in a log, RabbitMQ removes acknowledged messages. Replaying events for data migration or bug fixes requires republishing from the outbox table or a separate replay mechanism. Mitigation: outbox records are retained for a configurable period (default 7 days) before deletion; a replay endpoint exists for authorized admin use.

### Neutral
- **Team must learn message broker patterns** — developers need to understand consumer acknowledgements, prefetch counts, retry policies, dead-letter exchanges, and queue binding. Training and code templates are required.
- **Testing event-driven flows is more complex** — unit tests can verify the OutboxProcessor publishes events, but integration tests must verify that consumers receive and process events correctly. Testing eventual consistency scenarios requires test timeouts and polling. The test harness includes a Docker Compose RabbitMQ instance for integration tests.
