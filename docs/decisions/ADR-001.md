# ADR-001: Architecture Style

- **Status:** Accepted
- **Date:** 2026-06-20
- **Deciders:** Architecture Team

## Context

The Telecom OSS/BSS Platform must support 19 bounded contexts spanning CRM, product catalog, ordering, subscription management, rating, billing, invoicing, payments, collections, provisioning, network inventory, ticketing, workflow, notifications, reporting, audit, and IAM.

The architecture must support:
- Clean separation of business domains with well-defined boundaries
- Independent deployability of modules when scaling demands it
- Team autonomy as the organization grows
- Testability of domain logic without infrastructure dependencies
- Long-term maintainability (platform expected lifespan: 10+ years)

Three architectural styles were evaluated:

1. **Microservices** — Each bounded context deployed as an independent service with its own process, database, and deployment pipeline.
2. **Modular Monolith** — All bounded contexts deployed as a single process with strictly separated modules (Clean Architecture layers, per-module database schemas, in-process communication via MediatR).
3. **Layered Architecture** — Traditional N-tier architecture (Presentation → Business Logic → Data Access) with no explicit domain model or bounded contexts.

## Decision

We will adopt **Modular Monolith with Clean Architecture and Domain-Driven Design**.

Key specifics:
- All 19 bounded contexts run in a single process but are organized into separate .NET assemblies (Domain, Application, Infrastructure, Presentation per module).
- Communication between modules within the process uses MediatR (commands, queries, domain events).
- Cross-module state changes use asynchronous integration events via RabbitMQ (Outbox/Inbox patterns).
- Each module owns its database schema with zero cross-schema access.
- Modules follow Clean Architecture dependency inversion: Domain → Application → Infrastructure → Presentation.
- Extraction to microservices is a deployment-time decision triggered by proven scaling bottlenecks, not an architecture-time decision.

## Consequences

### Positive
- **Simplified development workflow** — single solution, single debug session, single deployment pipeline during initial development. Developers can work on multiple modules without context-switching between service deployments.
- **Free and nearly instant in-process communication** — MediatR command/query dispatch avoids network latency, serialization overhead, and transient failure handling needed in distributed systems.
- **Atomic cross-module transactions where needed** — within a single process, a database transaction can span multiple module schemas if a business process requires it (rare but possible).
- **Deferred distribution complexity** — distributed systems failures (network partitions, latency spikes, serialization mismatches, distributed tracing gaps) are avoided until there is proven need to extract a module.
- **Architecture test enforcement** — dependency rules, module boundary violations, and circular references can be caught at compile time with NetArchTest.
- **Low operational overhead** — one process to monitor, one set of health checks, one deployment artifact.

### Negative
- **Single process resource ceiling** — all modules share the same CPU and memory. If one module becomes resource-intensive (e.g., batch rating processing), it can impact the responsiveness of other modules. Mitigation: separate background processes for heavy workloads, configurable thread pool isolation.
- **No independent scalability** — you cannot scale the Rating module independently from the CRM module. Mitigation: extraction criteria trigger conversion to a microservice when a module shows sustained resource pressure.
- **Single point of failure** — a fatal bug in any module crashes the entire process. Mitigation: structured exception handling, health checks, process-level supervision (Docker restart policy, systemd).
- **Tighter coupling risk** — without the friction of network boundaries, developers may be tempted to create in-process coupling between modules. Mitigation: architecture tests enforce that cross-module communication uses events only — no direct references to another module's Application or Infrastructure layer.
- **Deployment coordination** — a deployment affects all modules simultaneously, even if only one module changed. Mitigation: feature flags for gradual rollout; lightweight canary deployments via multiple instances with load balancer traffic shifting.

### Neutral
- **Build tooling must enforce module boundaries** — compile-time checks (NetArchTest) and code analysis rules are essential to prevent boundary erosion over time.
- **Migration path to microservices is defined but unproven** — the extraction plan exists in the architecture document but has not been executed. Each module's Clean Architecture structure makes extraction technically possible but the operational experience of running extracted services will surface unknown challenges.
