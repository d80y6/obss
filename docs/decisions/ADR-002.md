# ADR-002: Multi-Tenant Strategy

- **Status:** Accepted
- **Date:** 2026-06-20
- **Deciders:** Architecture Team

## Context

The OSS/BSS Platform must serve multiple tenants (ISPs, business units, resellers) with strict data isolation. Each tenant's data must be invisible to all other tenants. The platform may eventually be offered as a SaaS product where multiple CSPs share a single deployment.

Three multi-tenant isolation strategies were evaluated:

1. **Database per Tenant** — Each tenant gets its own PostgreSQL database. Connection routing is determined at the application or connection-pool level.
2. **Schema per Tenant** — Each tenant gets its own schema within a shared PostgreSQL database. Tables are duplicated per schema.
3. **Shared Schema with TenantId Column** — All tenants share the same tables. A `TenantId` column on every tenant-scoped row distinguishes ownership.

Additional considerations:
- Row-Level Security (RLS) in PostgreSQL can enforce tenant isolation at the database level as a defense-in-depth layer.
- EF Core global query filters can automatically append `TenantId` to every query.
- Tenant provisioning must be fast (sub-second for new tenant creation).
- Schema migrations must be manageable across potentially hundreds of tenants.

## Decision

We will use **Shared Schema with TenantId column + PostgreSQL Row-Level Security (RLS)**.

Key specifics:
- Every tenant-scoped table includes a `TenantId` column (type `UUID`, non-nullable, indexed).
- EF Core global query filters (`HasQueryFilter`) automatically append `TenantId = @tenantId` to every LINQ query on tenant-scoped entities.
- PostgreSQL RLS is enabled on every tenant-scoped table with a policy that enforces `tenant_id = current_setting('app.tenant_id')::uuid`.
- The `TenantId` value is extracted from the JWT access token at the API Gateway and propagated through the request pipeline via an `ITenantContext` scoped service.
- Tenant provisioning is handled by the IAM module: creating the tenant record in `iam.tenants`, seeding default roles and permissions, and creating the initial admin user.
- No per-tenant schema creation or database creation is needed — new tenants simply get a row in the shared tables.

## Consequences

### Positive
- **Fast tenant provisioning** — creating a new tenant is a single row insert. No database creation, no schema cloning, no migration execution. Tenant provisioning completes in under 100ms.
- **Simplest operations** — one database to back up, one set of migrations to run, one connection pool to manage. Schema changes affect all tenants simultaneously with no per-tenant migration orchestration.
- **Lowest resource cost per tenant** — no per-tenant database overhead (connection slots, shared buffers, WAL amplification). Marginal cost per additional tenant approaches zero at the database level.
- **EF Core global query filters provide compile-time safety** — developers cannot accidentally omit a `Where(TenantId == ...)` clause because the filter is applied automatically by the ORM. Integration tests verify isolation.
- **PostgreSQL RLS provides defense-in-depth** — even if the application layer incorrectly queries without a TenantId filter, the database row-level security policy blocks cross-tenant data access. RLS policy cannot be bypassed by the application connection.
- **Straightforward cross-tenant queries for super-admins** — platform-level operations (cross-tenant reporting, tenant comparison) are possible by setting the RLS policy to allow all for super-admin roles.
- **Standard tooling compatibility** — pgAdmin, DataGrip, and migration tools work without per-tenant configuration.

### Negative
- **No per-tenant data separation** — a bug in a migration that drops a column affects all tenants simultaneously. There is no tenant-level isolation for schema changes. Mitigation: migrations are tested in CI with representative data volumes; blue-green deployment pattern reduces migration risk.
- **RLS performance overhead** — every query on a tenant-scoped table has an additional RLS policy evaluation. The overhead is typically <5% but can increase on tables with complex policies. Mitigation: RLS policies are kept simple (single column equality check); tables with the highest query volume are monitored for RLS overhead.
- **No per-tenant backup/restore granularity** — a single tenant's data cannot be restored independently without restoring the entire database. Mitigation: logical data export per tenant for customer-facing data recovery scenarios; point-in-time recovery at the database level for disaster scenarios.
- **TenantId must be present on every query** — if any code path omits the TenantId filter and RLS is misconfigured, data leakage is possible. Mitigation: RLS is the safety net; integration tests explicitly verify cross-tenant isolation.
- **Connection pooling per database (not per tenant)** — a single tenant's heavy query load can consume connection pool resources and impact other tenants. Mitigation: statement timeout limits, connection pool sizing, and per-tenant resource monitoring.
- **Index size grows with all tenants' data** — unlike schema-per-tenant where indexes are per-tenant, shared-schema indexes span all tenants. A single tenant's data growth affects index scan performance for all tenants. Mitigation: partitioned tables by TenantId for large tables (invoices, usage records); regular VACUUM and maintenance.

### Neutral
- **RLS requires careful configuration** — policies must be created for every tenant-scoped table. New tables added in migrations must have RLS enabled as part of the migration script. CI gates check for missing RLS policies.
- **Tenant context propagation must be reliable** — the `TenantId` claim must be present in every authenticated request. Background jobs must explicitly set the tenant context. A tenant context propagation failure could result in missing TenantId (which violates the NOT NULL constraint) or incorrect TenantId (which RLS would catch).
