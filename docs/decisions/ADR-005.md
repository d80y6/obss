# ADR-005: Database Strategy

- **Status:** Accepted
- **Date:** 2026-06-20
- **Deciders:** Architecture Team

## Context

The OSS/BSS Platform must persist data for 19 bounded contexts, each with different storage characteristics:
- **IAM, CRM, ProductCatalog, Orders, Subscriptions** — highly relational, transactionally consistent, moderate read/write volume.
- **Rating, Billing, Invoices** — high write throughput during billing cycles, large batch operations, monthly bill-run processing of 100K+ records.
- **Provisioning, Workflow, Ticketing** — moderate volume, strong consistency for state machines.
- **Reporting** — complex analytical queries across multiple module schemas, materialized views, scheduled report generation.
- **Audit** — append-only write-heavy table with long retention, immutable log.
- **NetworkInventory** — hierarchical/tree data (network topology), IP pool management with range queries.
- **Notifications** — high-write throughput for delivery logs with short retention.

Additional considerations:
- Multi-tenant data isolation (see ADR-002)
- Event-driven architecture requires Outbox and Inbox tables per module
- Yemen market requires on-premise deployment capability
- Search across customer data, tickets, and inventory items
- Caching for frequently accessed data (product catalog, tenant settings, user sessions)

Three database strategies were evaluated:
1. **Single monolithic PostgreSQL database** — one database, all modules share same instance but use separate schemas.
2. **Multiple PostgreSQL databases** — one database per module, separate connection pools, cross-module queries via API.
3. **Polyglot persistence** — different databases for different workloads (PostgreSQL for relational, MongoDB for catalog, Neo4j for network topology, Elasticsearch for search).

## Decision

We will use **PostgreSQL for all modules (one database, per-module schemas), Redis for caching, and OpenSearch for search and analytics**.

Key specifics:
- **PostgreSQL (Primary)**: Single PostgreSQL 16+ database instance. Each module gets its own schema (e.g., `iam`, `crm`, `catalog`, `orders`, `billing`, `rating`, `invoices`, `payments`, `provisioning`, `inventory`, `network`, `workflow`, `ticketing`, `notifications`, `reporting`, `audit`). Outbox and Inbox tables live in each module's schema.
  - EF Core (Npgsql provider) for write operations with domain aggregates.
  - Dapper for read-side queries and reporting (optimized SQL, no ORM overhead).
  - PostgreSQL Row-Level Security for tenant isolation.
  - Partitioning for large tables: `invoices.invoices` by month, `audit.audit_logs` by month, `rating.usage_records` by month.
  - JSONB columns for flexible attributes and custom fields.
  - Full-text search (tsvector) for basic search needs.
- **Redis (Caching)**: Redis 7+ for distributed caching, session storage, rate limiting counters, and temporary data (short-lived tokens, email verification codes).
  - Cache-aside pattern for frequently accessed read models.
  - Distributed cache for the API Gateway (resolved permissions, rate limit counters).
  - Storage for background job locks (distributed mutex).
- **OpenSearch (Search & Analytics)**: OpenSearch 2+ for full-text search across customers, tickets, products, and orders. Also serves as the analytics engine for the Reporting module's dashboards and ad-hoc queries.
  - Data synchronized via CDC events (integration events consumed by a search indexer service).
  - Materialized views in PostgreSQL for operational reporting; OpenSearch for interactive search and analytics.

## Consequences

### Positive
- **Single PostgreSQL database simplifies operations** — one instance to back up, one point-in-time recovery target, one replication configuration, one set of monitoring. Per-module schemas provide logical separation without operational overhead.
- **Cross-module queries are possible (within limits)** — the Reporting module can query across schemas for data mart materialization. Super-admin tools can join across schemas for debugging. This would be impossible with per-module databases.
- **PostgreSQL maturity and reliability** — battle-tested for OLTP workloads, excellent JSONB support, full-text search, partitioning, GiST/GIN indexes for network range queries. Active community, regular releases, strong tooling (pgBackRest, pg_stat_statements, auto_explain).
- **Redis provides sub-millisecond cache lookups** — reduces database load for hot data (product catalog, tenant settings, user permissions). Session storage offloads database reads.
- **OpenSearch provides superior search** — full-text search with scoring, faceted search, autocomplete, and aggregations. The database remains the source of truth; OpenSearch is a derived index for search performance.
- **Polyglot complexity avoided** — introducing MongoDB, Neo4j, or other databases would require additional operational expertise, backup strategies, and learning curves. PostgreSQL with JSONB and partitioning handles most workloads well enough.
- **On-premise deployable** — all three components (PostgreSQL, Redis, OpenSearch) can run on-premise without cloud dependencies, critical for the Yemen market.

### Negative
- **Database is a single point of failure** — a PostgreSQL outage takes down all modules. Mitigation: streaming replication with automatic failover (Patroni/PgBouncer), connection pooling, read replicas for reporting queries. RTO target: <60 seconds for automatic failover.
- **No per-module scalability** — a write-heavy module (Rating during bill run) can impact the performance of other modules sharing the same database instance. Mitigation: resource limits per connection (statement timeout, work_mem), separate connection pools per module with `pgbouncer`, query monitoring via `pg_stat_activity`.
- **OpenSearch is another infrastructure component** — adds operational overhead (cluster management, index lifecycle management, snapshot/restore). Mitigation: single-node OpenSearch for MVP, multi-node cluster for production. Index lifecycle policies automate rollover and retention.
- **Read replicas introduce eventual consistency for reporting** — if the Reporting module reads from a replica, there is replication lag. Mitigation: reporting on replicas is acceptable for approximate analytics; operational reporting that needs current data reads from the primary with mark-for-read-only connections.
- **Single database backup is large** — backing up a single large PostgreSQL database takes longer than backing up multiple smaller databases. Mitigation: pgBackRest with incremental backup, parallel backup, and S3-compatible object store target.
- **PostgreSQL full-text search is limited** — while adequate for simple search, it lacks the relevance scoring, typo tolerance, and faceted search that OpenSearch provides. The decision to add OpenSearch acknowledges this limitation.

### Neutral
- **EF Core + Dapper dual access pattern** — the team must be comfortable with two data access approaches. EF Core for aggregate-centric writes (with change tracking, concurrency handling, global query filters). Dapper for read-optimized queries (raw SQL, hand-tuned, no overhead). This is consistent with the CQRS pattern.
- **Schema migrations are shared** — since all modules share one database, migrations run sequentially. A long-running migration blocks all subsequent migrations. Mitigation: migrations are designed to be backward-compatible (additive changes only); wrappers (DbContext per module) allow parallel schema migrations within a single module without affecting others.
- **OpenSearch indexing introduces latency** — data written to PostgreSQL is not immediately available in OpenSearch. The indexing pipeline (CDC → Integration Event → Search Indexer) introduces seconds of delay. This is acceptable for search but must be documented for stakeholders.
