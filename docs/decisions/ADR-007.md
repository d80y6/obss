# ADR-007: Deployment Strategy

- **Status:** Accepted
- **Date:** 2026-06-20
- **Deciders:** Architecture Team

## Context

The OSS/BSS Platform must be deployed in the Yemen market where internet reliability, power stability, and cloud service availability are inconsistent. The deployment strategy must balance development velocity (rapid iteration during MVP) with production operational requirements (high availability, disaster recovery, scaling).

The platform consists of:
- .NET 9 modular monolith application (API Gateway + 19 modules in one process, plus background workers)
- PostgreSQL 16+ database (primary data store)
- Redis 7+ (caching, session storage, rate limiting)
- RabbitMQ (message broker)
- OpenSearch 2+ (search and analytics)
- Keycloak (identity provider)
- Reverse proxy / load balancer (nginx or HAProxy)

Two deployment approaches were evaluated:
1. **Kubernetes (K8s)** — Container orchestration with pod autoscaling, service discovery, rolling updates, secrets management, and declarative infrastructure. Either self-managed (kubeadm, Rancher) or single-node (K3s, MicroK8s) for on-premise.
2. **Docker Compose** — Single-host container orchestration with Docker Compose. Simple deployment, low resource footprint, minimal operational overhead. Suitable for small-scale deployments and development environments.

## Decision

We will use **Docker Compose for development and MVP, but architect the application for future K8s migration from day one**.

Key specifics:
- **Development and MVP Phase**: Docker Compose runs on a single Linux VM (or bare-metal). All services are defined in a `docker-compose.yml` with health checks, dependency ordering, volume mounts for persistent data, and environment-specific configuration files.
- **Production Target Architecture**: Kubernetes (K3s for single-node on-premise, or managed K8s for future cloud deployment). The modular monolith is packaged as a single container image but with configuration that supports decomposing into multiple deployments (API Gateway + modules, background workers) without code changes.
- **K8s-Ready Design Principles**:
  - All application configuration is environment-injected (environment variables, not config files committed to repo).
  - Health check endpoints (`/health`, `/ready`) are implemented from day one.
  - Structured logging to stdout (JSON format) — no file-based logging.
  - Stateless application containers — all stateful data in PostgreSQL, Redis, RabbitMQ.
  - Graceful shutdown handling (`IHostedService` stop, in-flight request completion).
  - OpenTelemetry metrics and traces for pod-level monitoring.
- **Containerization**: Multi-stage Docker builds with `mcr.microsoft.com/dotnet/sdk:9.0` for build and `mcr.microsoft.com/dotnet/aspnet:9.0` for runtime. Image size target: <200 MB. Distroless base image for production.
- **CI/CD Pipeline**: GitHub Actions build and push container images to a container registry. Docker Compose deployments pull images and restart services. Future: ArgoCD or Flux for K8s GitOps deployments.

## Consequences

### Positive
- **Docker Compose is simple and fast to set up** — a developer can have the entire platform running locally in under 5 minutes (after initial image build). No K8s cluster, no kubectl context switching, no Helm charts needed during development.
- **Low infrastructure requirements for MVP** — a single VM with 8 CPU cores, 16 GB RAM, and 100 GB SSD can run the entire platform stack for small-scale trials (up to 1000 subscribers). This is achievable in the Yemen market where dedicated server hardware may be limited.
- **K8s migration path is low-risk** — since the application is containerized, health-checked, stateless, and configuration-driven, migrating from Docker Compose to K8s requires only infrastructure changes (deployments, services, ingresses, ConfigMaps) — not application changes. The same container image runs in both environments.
- **Horizontal scaling is possible even in Docker Compose** — the modular monolith container can be scaled `docker-compose up --scale app=3` behind a reverse proxy (nginx). The stateful backing services (PostgreSQL, Redis, RabbitMQ) are the scaling bottleneck, not the application.
- **Consistent environments from development to production** — the same `Dockerfile` and `docker-compose.yml` are used across local dev, CI, staging, and production. Environment-specific differences are limited to configuration values and resource limits.
- **Simpler disaster recovery** — Docker Compose backups involve dumping PostgreSQL, Redis RDB, and backing up configuration. Less complex than K8s etcd backup + PVC snapshots + K8s manifest restoration.

### Negative
- **No built-in orchestration features** — Docker Compose does not provide automated pod rescheduling, rolling updates with health check gating, auto-scaling, or self-healing. If the application container crashes, Docker Compose restarts it (restart: unless-stopped) but with no sophisticated health-based decisions. Mitigation: process-level supervision via Docker restart policies; external monitoring (Prometheus + Alertmanager) for notification of failures; manual intervention required for node failures.
- **Single-host bottleneck** — all services run on one machine. Resource contention between the application, PostgreSQL, Redis, RabbitMQ, and OpenSearch can impact performance. A memory leak in one service affects all others. Mitigation: resource limits (mem_limit, cpus) in Docker Compose; separate backing services to dedicated VMs for production; monitoring dashboards for resource utilization.
- **No zero-downtime deployments** — Docker Compose does not support rolling updates with health check gating. The standard approach is `docker-compose up -d --force-recreate` which briefly stops and restarts containers. Mitigation: blue-green deployment script that starts a new stack alongside the old one, then switches the reverse proxy traffic; acceptable for MVP where scheduled downtime windows are feasible.
- **No native service mesh** — Kubernetes provides service mesh integration (Istio, Linkerd) for mTLS, traffic splitting, and observability. Docker Compose lacks this. Mitigation: mTLS is not required within a single host (network isolation is via Docker networks); observability is handled by the application-level OpenTelemetry instrumentation.
- **No built-in secrets management** — Docker Compose supports environment files but does not have native secrets encryption. Mitigation: HashiCorp Vault agent sidecar for production secrets; `.env` files used only in local development (never committed to git).
- **Stateful services are harder to operate without K8s Operators** — PostgreSQL (Patroni), RabbitMQ, and OpenSearch all have K8s Operators that automate cluster management, failover, and backup. Without K8s, these must be managed manually or with custom scripts. Mitigation: Docker Compose configurations for single-instance stateful services; dedicated VM for PostgreSQL with Patroni + pgBackRest for HA; RabbitMQ mirrored queues with manual failover procedures.

### Neutral
- **Network architecture is simpler without K8s** — Docker Compose uses internal Docker networks with container name-based DNS resolution. No need for K8s Services, Ingress, or NetworkPolicies for the MVP. The API Gateway listens on port 443 (via reverse proxy) and routes to internal containers by name.
- **Observability stack still works** — Prometheus (scraping Docker container metrics + application /metrics endpoints), Grafana dashboards, and Loki for log aggregation all run in Docker Compose. The monitoring stack is the same regardless of deployment target.
- **Team must learn both Docker Compose and Kubernetes** — developers operate in Docker Compose daily but must understand K8s concepts (pods, deployments, services, configmaps) to ensure the application remains K8s-compatible. The CI/CD pipeline produces K8s manifests alongside Docker Compose files from the same source of truth.
